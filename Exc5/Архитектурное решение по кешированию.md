План по внедрению кэширования в систему «Александрит».

---

## 1. Анализ системы и выбор области для кеширования

**Основная проблема**: операторы в MES жалуются на низкую скорость загрузки страниц.
- MES UI показывает большие списки заказов (с различными статусами), и при большом числе заказов запросы к базе (MES DB) отрабатывают медленно.
- Также новые B2B-клиенты хотят быстрее получать информацию о заказах через MES API (скорость ответа на REST-запросы важна для интеграции).

**Вывод**: наибольший выигрыш даст кеширование данных в MES. Самый «тяжёлый» сценарий — **получение списка заказов** (по статусу, по оператору, по дате). При этом записи (обновления статуса) относительно редки, по сравнению с чтениями операторов.

---

## 2. Мотивация

**Почему нужно кеширование**
1. **Скорость отображения списков заказов**:
    - Операторы должны быстро видеть новые заказы и брать их в работу. Если запрос формируется напрямую из БД при каждом обращении, и заказов много, время ответа растёт.
2. **Снижение нагрузки на базу данных MES**:
    - Рост количества B2B-заказов и параллельная работа десятков операторов приводят к тому, что MES DB становится «узким местом». Кеширование позволит «сгладить» пики обращений.
3. **Лучший пользовательский опыт**:
    - Операторы получат минимальную задержку при открытии страницы в MES UI, что повысит их продуктивность и ускорит обработку заказов в производстве.

**Какие элементы системы планируем кешировать**
- **Списки заказов** (например, все заказы со статусом `PRICE_CALCULATED`, `MANUFACTURING_APPROVED`, `MANUFACTURING_STARTED` и т.д.).
- **Возможно, отдельные справочные данные**.

Основной фокус — **кешировать результат запросов на чтение** в MES API, чтобы ускорить отдачу данных для MES UI и интеграций (B2B-клиенты).

---

## 3. Предлагаемое решение

### 3.1. Тип кеширования: серверное

- **Почему серверное**:
    - Клиентское (браузерное) кеширование сложно контролировать, особенно при работе с несколькими операторами, которые могут видеть разные статусы.
    - Серверное кеширование даёт нам полный контроль над данными, условиями инвалидации и обновления.

### 3.2. Паттерн кеширования

Варианты паттернов:

1. **Cache-Aside**
    - При чтении MES API проверяет кеш. Если данных нет, идёт в БД, после чего кладёт результат в кеш. При записи данных в БД инвалидация (удаление или обновление) происходит по ключу, связанному с этими данными.
    - **Плюсы**: простой в реализации, даёт гибкость управления кешем.
    - **Минусы**: при первом запросе («промахе» в кеше) пользователь всё равно ждёт, пока данные придут из БД.

2. **Write-Through**
    - При записи данные мгновенно обновляются как в БД, так и в кеше. При чтении мы всегда можем взять актуальные данные из кеша.
    - **Плюсы**: уменьшает вероятность «промахов», так как кеш всегда актуален после записи.
    - **Минусы**: сложнее поддерживать一致ность, выше нагрузка на кеш при частых записях.

3. **Refresh-Ahead**
    - Приложение проактивно «разогревает» кеш, предугадывая, какие данные скоро понадобятся.
    - **Плюсы**: при обращении к данным они уже в кеше.
    - **Минусы**: сложнее прогнозировать, какие списки заказов оператор захочет увидеть. При ошибочном прогнозе можно тратить ресурсы впустую.

**Выбор**:
- Опираясь на то, что записи (смена статуса) в MES происходят реже, а чтение (списки заказов) — гораздо чаще, оптимальным будет **Cache-Aside**.
- Причина:
    - Простота внедрения (чтение → «промах» → БД → кеш; запись → БД → инвалидация кеша).
    - При редких записях нет смысла усложнять Write-Through, и Refresh-Ahead не даёт существенных преимуществ без точных прогнозов.

**Пояснения**:
- При чтении списка заказов для конкретного статуса (например, `PRICE_CALCULATED`) MES API сначала обращается к кеше. При отсутствии данных идёт в БД, сохраняет полученный результат в кеше, а затем возвращает клиенту.
- При обновлении статуса в БД кеш по соответствующему ключу удаляется (или становится неактуальным). На следующем запросе список будет заново загружен из БД и положен в кеш.

## 5. Стратегия инвалидации кеша

### 5.1. Предлагаемый подход

- **Инвалидация по ключу (программная)**:
    - Когда статус заказа меняется, MES API понимает, что заказ «вышел» из одного статуса и перешёл в другой. Значит, нужно удалить из кеша соответствующий ключ.
    - Например, если заказ был в `PRICE_CALCULATED`, мы удаляем ключ `OrderList:PRICE_CALCULATED`, чтобы при следующем чтении данные пересчитались.

- **Дополнительно — временная (TTL)**
    - Можно назначить небольшое время жизни (например, 2–5 минут) на кешируемый список, чтобы гарантированно обновлять данные, даже если ручная инвалидация не сработала (в случае ошибок).

**Почему это подходит**:
- MES логика статусов достаточно предсказуема, мы знаем, из какого статуса в какой переходим. Значит, можем легко вычислить, какие кеш-ключи становятся «устаревшими».
- Простое решение даёт хороший баланс между актуальностью и производительностью.

### 5.2. Почему не подходят другие варианты

- **Полностью временная (TTL) без ручной инвалидации**: может приводить к тому, что оператор увидит устаревший список (придётся ждать, пока истечёт TTL).
- **Глобальная очистка всего кеша** при любой записи — неэффективно: нарушает идею избирательного кеширования.
- **Частичное обновление** (например, автоматическая замена отдельной записи в списке) возможно, но усложняет код. Проще и надёжнее инвалидировать весь набор заказов данного статуса, чтобы не ошибиться.

## 6. Сравнительный анализ стратегий

| **Стратегия**              | **Преимущества**                                                                 | **Недостатки**                                                                |
|----------------------------|----------------------------------------------------------------------------------|-------------------------------------------------------------------------------|
| **Cache-Aside** (выбран)  | - Лёгкая реализация<br>- Хорош, когда больше чтений, чем записей<br>- Гибко управляется в коде | - Нужно обрабатывать «промах» и вручную инвалидировать<br>- При первом запросе (miss) всё же идём в БД |
| **Write-Through**          | - Данные сразу есть в кеше, нет «miss» после записи<br>- Меньше расхождений      | - Усложнённая логика записи (запись в БД + кеш одновременно)<br>- Для MES с редкими записями может быть избыточным |
| **Refresh-Ahead**          | - «Разогревает» кеш заранее<br>- Полезно при прогнозируемых запросах             | - Нужно предсказывать, что будет нужно<br>- Ошибочный «прогрев» ведёт к расходам впустую |

**Итог**:
- **Cache-Aside** лучше всего балансирует простоту реализации и выигрыш в производительности при сценарии MES, где чтений много, а запись статусов относительно редка.

---

Таким образом, данное решение поможет решить основную проблему низкой скорости MES, повысить качество обслуживания B2B и B2C-пользователей, а также улучшить производительность всей системы.

План по внедрению трейсинга в систему «Александрит».

---

## 1. Анализ системы в контексте планирования трейсинга

Основные места, где заказ может «сломаться» или «зависнуть»:
1. **Передача заказа из Internet Shop в CRM**.
    - Через очередь RabbitMQ или напрямую через Shop DB. Возможны случаи, когда сообщение о новом заказе теряется или обрабатывается с большой задержкой.
2. **Статус заказа между CRM и MES**.
    - MES может не получить вовремя сообщение от CRM (или наоборот).
    - При расчёте стоимости в MES — процесс может «зависнуть» на долгую обработку 3D-файла.
3. **Обновление статусов в CRM**.
    - После того как MES завершил производство, заказ переходит в PACKAGING/SHIPPED, в CRM — MANUFACTURING_COMPLETED/CLOSED. Если сообщение потерялось, в CRM может остаться «подвешенный» заказ.
4. **B2B-заказы через API MES**.
    - Сторонние клиенты вызывают MES API, но заказ фактически регистрируется в CRM через очередь. Если этот кусок логики сбоит, заказ может не появиться в CRM.
5. **Хранение и передача 3D-файлов**.
    - При потере связи с S3 или при ошибке загрузки MES не сможет рассчитать стоимость — заказ «стопорится».

### Данные, которые должны попадать в трейсинг

Чтобы точно понять, где «застрял» заказ, нам нужны:

1. **Уникальный идентификатор (trace-id + span-id)**: генерируется при создании заказа (или при первом запросе от B2C/B2B) и передаётся по всем системам.
2. **Имя операции** (например, `POST /shopApi/orders`, `CalculatePrice`, `UpdateStatus`, `B2B/CreateOrder`).
3. **Метаданные о времени**:
    - Время входа в сервис (startTimestamp)
    - Время выхода из сервиса (endTimestamp)
    - Продолжительность (duration)
4. **Статус выполнения** (успешно / с ошибкой).
5. **Теги/лейблы (labels)**, чтобы потом находить конкретные треки:
    - Идентификатор заказа (orderId)
    - Тип заказа (B2C/B2B)
    - Текущий статус заказа (INITIATED, PRICE_CALCULATED, MANUFACTURING_STARTED и т.д.)
    - Идентификатор пользователя / клиента (customerId) — с учётом GDPR/конфиденциальности, хранить только технический id.
    - Ссылка на 3D-файл (при необходимости) или его hash.

Такие данные позволят проследить «путь» заказа и увидеть, где он остановился или почему возникли задержки.

## 2. Мотивация (зачем нужен трейсинг)

Без трейсинга мы видим только «кусочные» логи или метрики, но не можем собрать цельную картину прохождения одного заказа через все микросервисы. Распределённый трейсинг даёт бизнесу и разработчикам:

1. **Быстрое нахождение «зависаний» и точное определение точки сбоя**
    - Сокращает время поиска причин, когда B2B или B2C-заказ не дошёл до CRM, или MES «забыл» его производить.

2. **Повышение прозрачности для SLA**
    - Позволяет понять, на каком этапе (Shop/CRM/MES) появились задержки и кто несёт ответственность.
    - Бизнес может выставлять SLA-обязательства перед партнёрами, опираясь на точные данные о времени обработки.

3. **Улучшение метрик выполнения (Lead Time, Cycle Time)**
    - Например, время от SUBMITTED до PRICE_CALCULATED, от PRICE_CALCULATED до SHIPPED.
    - Позволяет бизнесу оптимизировать цепочку поставки, повышать пропускную способность, точнее планировать производство.

4. **Влияние на технические и бизнес-метрики**
    - **Mean Time To Repair (MTTR)**: с трейсингом команды быстрее находят сбои, время восстановления сокращается.
    - **Количество жалоб**: меньше недовольных клиентов, потому что проблема решается быстрее (или даже предотвращается).
    - **Ускорение интеграций**: внешние партнёры видят прозрачность процессов, выше доверие к API.

## 3. Предлагаемое решение

### 3.1. Технологии и компоненты

1. **Выбор системы трейсинга**:
    - Наиболее популярен **Jaeger**.
    - Если нужно гибко интегрировать Java, .NET, RabbitMQ и фронтенды на JS, рекомендуем OpenTelemetry + Jaeger, так как OpenTelemetry охватывает широкий список SDK.

2. **Добавление `Tracing Collector / Jaeger Collector`** (новый сервис)
    - Все микросервисы (Shop API, CRM API, MES API и даже фронтенды) будут отправлять данные в этот Collector.

3. **Агент / библиотека OpenTelemetry**
    - Для **Spring Boot (Java)** есть готовые starter’ы (opentelemetry-instrumentation-spring, micrometer и т.д.).
    - Для **.NET/C#** (MES API) также имеются пакеты OpenTelemetry.
    - На фронтендах (Vue, React) можно включить клиентские трейсинги, но это второстепенно. В первую очередь важны бэкенды.

4. **Traces Storage + UI**
    - Jaeger имеет собственное хранилище (через Cassandra, ES или просто in-memory для тестов) и веб-интерфейс (UI) для просмотра.
    - Можно развернуть это в том же Yandex Cloud.

5. **Интеграция с RabbitMQ**
    - Важно передавать trace-id и span-id в заголовках сообщений.
    - На стороне потребителя (MES, CRM) нужно считывать эти заголовки и продолжать трейс.

### 3.3. Основные шаги внедрения

1. **Подключить OpenTelemetry к каждому сервису**:
    - Shop API (Java/Spring), CRM API (Java/Spring), MES API (C#).
    - При входящих HTTP-запросах считывать заголовки `traceparent`, `tracestate` (W3C Trace Context) или `x-b3-*` (Zipkin/B3), продолжать существующий trace.
    - При исходящих HTTP-запросах или отправке сообщений в RabbitMQ добавлять заголовки с trace-id.
2. **Настроить RabbitMQ-интеграцию**:
    - При получении сообщения в MES или CRM извлекать trace-id, создавать child-span, логировать событие «Received message from RabbitMQ».
    - При отправке сообщения (publish) — создавать span, указывая, какой заказ, какой routing key, итд.
3. **Развернуть Jaeger Collector и UI** (в DevOps-контурах):
    - Можно в Yandex Cloud, либо on-premise.
    - Настроить retention (сколько дней/часов храним трейс-данные).
4. **Добавить конфигурацию для фильтрации чувствительных данных** (чтобы не логировать PII или конфиденциальную информацию о клиентах).
5. **Обучить команду** пользоваться Jaeger UI для быстрого поиска заказов по `orderId` и просмотра цепочки.

## 4. Компромиссы

1. **Сложность интеграции с проприетарными модулями**
    - MES API пока у нас есть исходный код, но если бы это была «чёрная коробка», то внедрить трейсинг было бы сложно (пришлось бы использовать прокси или адаптер).
2. **Дополнительная нагрузка на производительность**
    - При большом количестве запросов и сообщений объём трейс-данных может быть огромным. Нужно ограничивать (например, 10% или 1% запросов).
3. **Дорогостоящее хранение**
    - Если компания хочет хранить все трейс-данные продолжительное время, объёмы могут стать очень большими. Обычно трейсинг хранят короткий период (несколько дней), а полные логи — дольше.
4. **Не принесёт пользы, если сервисы не «пропускают» trace-id**
    - Трейсинг не сработает в связке, если хотя бы одно звено в цепочке не поддерживает передачу заголовков.
5. **Внедрение требует изменений в коде**
    - Нужно дорабатывать Shop API, CRM API, MES API, встраивая библиотеки OpenTelemetry, настраивая спаны. Это требует времени и ресурсов команды.

## 5. Аспекты безопасности

1. **Доступ к Jaeger UI**
    - Должен быть ограничен внутренней сетью (VPN или корпоративная сеть).
    - Включить **аутентификацию** (например, Basic Auth, OAuth2 Proxy) — чтобы только сотрудники с ролью «Поддержка» или «Разработчик» видели трейс-данные.
2. **Защита конфиденциальных данных**
    - В трейсах не должны передаваться личные данные клиентов (адреса, телефоны). Лучше только orderId, userId (ID клиента).
3. **Разделение прав**
    - Отдельный доступ на чтение (просмотр) и на администрирование Collector’а (перезапуск).
4. **Шифрование**
    - Связь между сервисами и Collector’ом желательно поднять поверх TLS, чтобы не передавать trace-id и метаданные в незашифрованном виде.

Внедрение распределённого трейсинга на базе OpenTelemetry + Jaeger позволит компании «Александрит»:

- Быстро находить «зависшие» заказы и исключить ситуацию, когда сообщения теряются без диагноза.
- Явно видеть цепочку обработки: Shop → RabbitMQ → CRM/MES → базы данных.
- Сократить время расследования инцидентов, тем самым повышая доступность и качество сервиса.
- Повысить доверие B2B-партнёров, благодаря прозрачности и доказуемому SLA.

Однако реализация требует дополнительных ресурсов, включая доработку кода в каждом сервисе, развёртывание и обслуживание трейсинг-инструментов, а также продуманную безопасность и политику хранения данных.
